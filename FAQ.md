# Frequently Asked Questions (FAQ)

This document answers the most common questions regarding the Compiler Construction Lab Project.

If your question is not answered here, consult the **Project Manual** first. If the answer is still unclear, contact the instructor before the submission deadline.

---

# Repository & GitHub

### 1. Why do we need to fork the instructor repository?

Forking creates your own copy of the project repository under your GitHub account while preserving a link to the original repository. All development should be carried out in your group's repository.

---

### 2. Should we clone the instructor repository?

No.

Clone **your own fork**, not the instructor repository.

---

### 3. Can we rename our repository?

Yes.

Use the naming convention specified in the project README.

---

### 4. Should our repository be public or private?

Keep the repository **Public**.

---

### 5. Can we delete our repository after submission?

No.

---

### 6. Can we reorganize the project directory structure?

Yes.

The suggested structure is a recommendation. You may organize your project differently if it remains clean, logical, and easy to navigate.

---

### 7. Should generated files be committed?

Generally, no.

Generated files, executables, object files, temporary files, and IDE configuration files should be excluded using the provided `.gitignore`.

---

# Team & Collaboration

### 8. Does every member need a GitHub account?

Yes.

Each student should contribute using their own GitHub account.

---

### 9. Can one member complete the entire project?

While work distribution is your group's responsibility, every member is expected to actively participate and understand the complete implementation.

GitHub contribution history may be reviewed during evaluation.

---

### 10. Do all members need to understand the entire project?

Yes.

During the presentation or viva, any member may be asked questions about any component of the project.

---

### 11. Can different members work on different modules?

Yes.

Dividing responsibilities is encouraged, but the final project should be well integrated and understood by every team member.

---

# Programming & Implementation

### 12. Which programming language should we use?

Use C or C++ together with Flex and Bison.

---

### 13. Can we use Windows?

Linux is strongly recommended.

Ubuntu or another Debian-based Linux distribution is preferred for development and testing.

---

### 14. Can we use WSL (Windows Subsystem for Linux)?

Yes.

WSL is acceptable if Flex, Bison, GCC, and Make work correctly.

---

### 15. Can we use an IDE?

Yes.

You may use any editor or IDE of your choice, such as:

- Visual Studio Code
- CLion
- Vim
- Emacs
- Sublime Text

---

### 16. Can we use external libraries?

Avoid unnecessary external libraries.

The project should primarily rely on standard C/C++ libraries together with Flex and Bison.

---

### 17. Can we modify the programming language specification?

No.

All mandatory requirements must follow the language specification described in the Project Manual.

---

### 18. Can we add extra features?

Yes.

Optional features may be implemented after all mandatory requirements have been completed.

---

# Git & Development

### 19. How often should we commit?

Commit regularly throughout the project.

Avoid making only one or two large commits near the deadline.

---

### 20. What makes a good commit message?

A good commit message clearly describes the change.

Example:

```
Implement symbol table insertion
```

Avoid vague messages such as:

```
update
fix
code
done
```

---

### 21. Should we create multiple branches?

Feature branches are recommended but not mandatory.

Small teams may work directly on the `main` branch if they coordinate carefully.

---

### 22. Can we rewrite Git history before submission?

Avoid rewriting published commit history unless absolutely necessary.

---

# AI Usage

### 23. Can we use ChatGPT, Claude, Gemini, or GitHub Copilot?

Yes.

AI tools may assist your learning and development.

---

### 24. Will using AI reduce our marks?

No.

Marks are based on your implementation quality and understanding, not on whether AI was used.

---

### 25. What happens if we cannot explain AI-generated code?

Failure to explain your own implementation during the demonstration or viva may result in mark deductions.

---

# Testing

### 26. What kinds of test cases should we include?

Your repository should include both valid and invalid programs demonstrating:

- Successful compilation
- Lexical errors
- Syntax errors
- Semantic errors

---

### 27. Should we include expected outputs?

Yes.

Providing expected outputs makes testing and evaluation easier.

---

# Documentation

### 28. Is documentation important?

Yes.

Documentation is part of the project evaluation.

Ensure that build instructions, execution instructions, and project documentation are complete and accurate.

---

### 29. Should our README explain how to build the project?

Yes.

Your project repository should contain clear build and execution instructions.

---

# Submission

### 30. What should we submit?

Submit only the GitHub repository link unless instructed otherwise.

---

### 31. Should we submit a ZIP file?

No.

ZIP files should only be submitted if explicitly requested.

---

### 32. Can we continue committing after the submission deadline?

Commits made after the deadline may not be considered during evaluation and may be treated according to the course submission policy.

---

### 33. What happens if our repository is incomplete?

Incomplete implementations may receive partial marks based on the completed components.

---

# Evaluation

### 34. Will GitHub activity be considered?

Yes.

Commit history and repository activity may be reviewed as part of the evaluation process.

---

### 35. Is a working compiler enough for full marks?

No.

Evaluation also considers documentation, code quality, repository organization, demonstration, presentation, and individual understanding.

---

### 36. What happens during the viva?

You may be asked to explain:

- Design decisions
- Grammar
- Data structures
- Algorithms
- Source code
- Error handling
- Testing strategy

Every member should be prepared to answer questions about the complete project.

---

# Getting Help

### 37. I found an error in the Project Manual. What should I do?

Report it to the instructor as early as possible.

Do not make assumptions if a requirement appears unclear.

---

### 38. When should we ask questions?

Ask questions well before the submission deadline.

Waiting until the final days may not leave enough time for clarification.

---

### 39. Where should project-related questions be asked?

Use the WhatsApp, Google Classroom, or email.

---

# Final Advice

Start early.

Commit regularly.

Test thoroughly.

Write clean, readable code.

Keep your repository organized.

Most importantly, understand everything your group submits.
